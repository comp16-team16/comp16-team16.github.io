10. Stores and State Management

Stores

Stores are used to maintain shared state in a way which ensures predictability and reduces side-effects. Each store consists of an immutable state object and a set of mutations which update the state. The state object itself is immutable, however can be updated by creating a copy of itself. This procedure is handled in createStore.

Assessment Store

This store provides a deterministic interface for communicating with the backend medical diagnosis model. It uses IndigoClient, which implements the underlying API.

Imports

import { createStore } from '../framework/framework.js'
import settings from '../settings.js'
import IndigoClient from '../assets/modules/IndigoClient.js'

Setup

Create a new store object:

const AssessmentStore = {
    state: {},
    mutations: {},
    actions: {}
}

We also need to create a new API client instance:

const client = new IndigoClient(settings.tokens.indigoAPIEndpoint)

State

Initially, all fields are set to null, since this store is not used until starting an assessment.

AssessmentStore.state.medicalConditionDistribution = null
AssessmentStore.state.setMedicalConditionNames = null
AssessmentStore.state.modalMedicalCondition = null

Mutations

Update the posterior distribution:

AssessmentStore.mutations.setMedicalConditionDistribution = function(state, distribution) {
    state.medicalConditionDistribution = distribution
}

Update the list of possible condition names. This should be called only once when starting the assessment.

AssessmentStore.mutations.setMedicalConditionNames = function(state, medicalConditionNames) {
    state.medicalConditionNames = medicalConditionNames
}

Update the most likely (modal) medical condition, as given by the backend model.

AssessmentStore.mutations.setModalMedicalCondition = function(state, modalMedicalCondition) {
    state.modalMedicalCondition = modalMedicalCondition

    console.log(state)
}

Actions

Since the API is asynchronous, we need to define several actions for this store.

This first action, initialises the medical condition distribution. It commits setMedicalConditionNames and setMedicalConditionDistribution.

// set the initial medical condition distribution.

AssessmentStore.actions.initMedicalConditionDistribution = function(context) {

    client
        .getPrior()
        .then(({ medicalConditionNames, medicalConditionPrior }) => {

            context.commit('setMedicalConditionDistribution', medicalConditionPrior)
            context.commit('setMedicalConditionNames', medicalConditionNames)
        })
}

Next, we define an action to update the posterior distribution once a new one is received from the model.

// change the medial condition distribution based on symptoms stated by
// patient.

AssessmentStore.actions.updateMedicalConditionDistribution = function(context, symptomsMap) {

    const priorDistribution = context.state.medicalConditionDistribution

    client
        .postPosterior(priorDistribution, symptomsMap)
        .then(({ posteriorDistribution, posteriorMode }) => {

            context.commit('setMedicalConditionDistribution', posteriorDistribution)
            context.commit('setModalMedicalCondition', posteriorMode)
        })
}

Exports

export default createStore(AssessmentStore)

Preferences Store

This store is used to hold user preferences. Currently, this just includes the app's theme (light or dark).

Imports

import { createStore } from '../framework/framework.js'
import settings from '../settings.js'

Setup

Create a new store object:

const PreferencesStore = {
    state: {},
    mutations: {}
}

State

Initially, the theme is set to whatever the default is in settings.js.

PreferencesStore.state.theme = settings.theme.default

Mutations

The only mutation is toggling/switching the current theme. The function below is used to toggle on the theme name using JavaScript's turnary operator.

function switchThemeName(currentThemeName) {
    return currentThemeName === 'light' ? 'dark' : 'light'
}

Now we just call this function whenever the switchTheme mutation is commited:

PreferencesStore.mutations.switchTheme = function(state) {
    state.theme = switchThemeName(state.theme)
}

Persistence

We want the user's theme choice to persist across browsing sessions.

Exports

Export the compiled store object.

export default createStore.persist('preferences', PreferencesStore)

User Store

This store holds metadata about the user, namely whether they've used the app before. This data is stored persistently on their local device.

Imports

import { createStore } from '../framework/framework.js'

Setup

const UserStore = {
    state: {},
    mutations: {}
}

State

Initially, the user's isFirstVisit state is set to true. This later changed to false after they've viewed the welcome slides.

UserStore.state.isFirstUse = true

Mutations

The only mutation needed is a settter for isFirstUse. Note that !!value ensures the updated state is a Boolean.

UserStore.mutations.negateFirstUse = function(state) {
    state.isFirstUse = false
}

Persistence

By definition, isFirstUse needs to persist accross browsing sessions.

Exports

export default createStore.persist('user', UserStore)
